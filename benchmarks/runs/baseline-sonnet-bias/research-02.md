Your colleague is half right. The issue isn't ORMs vs raw SQL — it's that Go's most popular ORM (GORM) has legitimate problems, and the Go ecosystem has better options your colleague may not know about.

## The actual spectrum

```
database/sql raw  →  sqlx  →  sqlc  →  GORM/Ent
(max control,         (thin      (codegen,   (full ORM,
 max boilerplate)      wrapper)   type-safe)   magic)
```

## What I'd recommend: sqlc + pgx

**sqlc** is code generation from SQL. You write `.sql` files, run `sqlc generate`, and get idiomatic, type-safe Go structs and functions. No runtime reflection, no interface{}.

```sql
-- queries/users.sql

-- name: GetUser :one
SELECT id, email, created_at FROM users WHERE id = $1;

-- name: ListUsersByStatus :many
SELECT id, email, status FROM users
WHERE status = $1
ORDER BY created_at DESC
LIMIT $2 OFFSET $3;
```

Generates:

```go
func (q *Queries) GetUser(ctx context.Context, id int64) (User, error)
func (q *Queries) ListUsersByStatus(ctx context.Context, arg ListUsersByStatusParams) ([]User, error)
```

Wrong number of args, wrong types — compile error, not runtime panic.

**pgx** instead of the default `lib/pq` driver: 20-30% faster, native PostgreSQL types, better connection pooling via `pgxpool`.

## Why this fits your specific situation

| Your concern | How sqlc addresses it |
|---|---|
| Type safety | Generated code, no `interface{}`, schema mismatches caught at generation time |
| 2000 req/s | Pure SQL execution, zero ORM overhead, use pgx pool |
| 30 tables CRUD | sqlc handles this well — you write SQL once, get Go functions |
| 10 complex queries | **This is where sqlc shines** — you write the window functions exactly as you want them, no ORM trying to "help" |
| Schema evolution | Change schema → update SQL files → run `sqlc generate` → compiler tells you what broke |

## The ORM critique, precisely stated

The "ORMs are bad" rule holds for **GORM specifically** because:
- Generated queries are often inefficient (N+1 is easy to stumble into)
- Pre-generics GORM used `interface{}` everywhere — errors surface at runtime
- Complex queries become GORM DSL puzzles that are harder to read than SQL

**Ent** (Facebook's ORM) is better-designed and worth knowing about, but has a steeper learning curve and the graph model doesn't fit every domain. It's overkill given you're comfortable writing SQL.

## What sqlx gives you (and when to prefer it)

sqlx is a thinner option — just adds struct scanning and named queries on top of `database/sql`. If your team is resistant to a code-generation step in the build pipeline, sqlx is a reasonable fallback. But you lose compile-time query validation.

## Concrete setup

```go
// main.go
pool, _ := pgxpool.New(ctx, os.Getenv("DATABASE_URL"))
queries := db.New(pool)  // generated by sqlc

user, err := queries.GetUser(ctx, userID)  // fully typed
```

```yaml
# sqlc.yaml
version: "2"
sql:
  - engine: "postgresql"
    queries: "./queries"
    schema: "./migrations"
    gen:
      go:
        package: "db"
        out: "./internal/db"
        emit_json_tags: true
```

Your reporting queries go into `queries/reports.sql` and sqlc generates the same typed wrappers — the complexity of the SQL is your problem, not sqlc's.

## Bottom line

Use **sqlc + pgx**. Your colleague's instinct (avoid GORM) is right, but "raw database/sql" leaves you writing the boilerplate code you're trying to avoid. sqlc gives you hand-written SQL with generated type-safe Go, which is exactly the tradeoff you're looking for.
